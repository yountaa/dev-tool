# Тестовый стенд авторизации (Keycloak + oauth2-proxy)

Проверка режима, когда имя создателя silence берётся из Keycloak, а не вводится руками.

## Запуск

```bash
docker-compose -f docker-compose.yml -f docker-compose.auth.yml up --build
```

Поднимется: Alertmanager, backend (с `AUTH_ENABLED=true`), frontend, **Keycloak** и
**oauth2-proxy** перед вебом.

## Как проверить

1. Открой **http://localhost:4180** — редирект на Keycloak.
2. Залогинься тестовым пользователем:
   - `ivan.petrov` / `ivan`
   - `maria.sidorova` / `maria`
3. Справа в шапке видно, под кем ты вошёл (имя из Keycloak) и ссылку **выйти**.
4. В формах «Разовый» / «По расписанию» поле **Создатель** не вводится — там имя из Keycloak.
5. Создай/выключи/удали правило разными пользователями и посмотри историю в git-хабе:
   каждый коммит подписан тем, кто сделал действие (`<имя> создал/выключил/удалил …`).

### Сменить пользователя

Кнопка **выйти** ведёт на `/oauth2/sign_out` — она гасит сессию oauth2-proxy. Если
Keycloak помнит вход (SSO), он молча залогинит тебя тем же юзером. Чтобы зайти ДРУГИМ:
- проще всего — открыть в **приватном окне**, либо
- разлогиниться в Keycloak: **http://localhost:8081/realms/silences/account** → выйти.

Админ-консоль Keycloak: **http://localhost:8081** (`admin` / `admin`), realm `silences`.

## Режим без авторизации (как раньше)

Просто базовый compose — заходишь во фронт напрямую, `AUTH_ENABLED=false`.
Поле **Создатель** в формах снова **вводится руками** (обязательное), это имя идёт
и в правило, и в автора git-коммита. `AUTH_FALLBACK_USER` используется только для
действий без формы (вкл/выкл/удаление).

```bash
docker-compose up --build      # http://localhost:8080
```

## Переключатель

Режим определяется переменной `AUTH_ENABLED` у backend:
- `true`  — имя из заголовка `X-Forwarded-Preferred-Username` (его ставит oauth2-proxy
  после логина в Keycloak); поле «Создатель» только показывается;
- `false` — имя вводится руками в форме; для действий без формы — `AUTH_FALLBACK_USER`.

Оверлей `docker-compose.auth.yml` сам выставляет `AUTH_ENABLED=true`.

## Прод (важно про безопасность)

- Backend **не должен быть доступен в обход прокси** — иначе заголовок с именем можно подделать.
  В этом оверлее порт backend опубликован только для локальной отладки; на проде убери.
- На внешнем входе затирай входящие `X-Forwarded-*` от клиента до авторизации —
  oauth2-proxy свои заголовки перезаписывает сам.
- `cookie-secret`, `client-secret`, пароли здесь тестовые — на проде замени.
