# dev-tool

Модульный веб-инструмент для эксплуатации. Сейчас один модуль — **silences**: ставит
алерты Alertmanager на silence (разово по датам и по расписанию), хранит правила в git
с историей и атрибуцией «кто что сделал». Бэкенд — FastAPI, фронт — Vue 3 + Vite,
разворачиваются как отдельные сервисы.

Архитектура: **1 модуль = 1 роутер (бэк) = 1 вкладка (фронт)**. Новый модуль добавляется
папкой в `backend/modules/` и `frontend/src/modules/` без правок в других местах.

---

## Возможности модуля silences

- **Разовый silence** — заглушить алерты на конкретный период по датам; ставится в AM сразу.
- **По расписанию** — повторяющиеся окна тишины (сетка дни×часы **+ минуты**); ставит шедулер.
- **Рабочие правила** — список локальных правил со статусом; включить/выключить, отредактировать
  (имя, метаданные, матчеры, окна), удалить.
- **Дубль-проверка** — при создании правила с теми же матчерами и концом алертинга запрос
  блокируется (409) с именем существующего правила.
- **История удалённых** — удалённое правило не стирается, а переносится в `<env>/old/...`
  (в вебе не показывается, в git остаётся).
- **Источник правды — git-хаб**, а не Alertmanager: список и статус считаются локально,
  в AM правила только применяются.
- **Авторизация (опционально)** — имя создателя берётся из Keycloak через oauth2-proxy;
  кто создал/изменил/удалил пишется автором git-коммита.

---

## Как это работает (поток данных)

```
веб ──► backend ──► git-хаб (хранилище правил, история)
                └──► Alertmanager (применение silence)
        шедулер (каждые SILENCE_CRON) ──► дочиняет недостающие silence по правилам
```

- Создание/правка пишет правило в git-хаб (файл `<env>/<kind>/<id>.json`) и применяет в AM.
- Шедулер периодически синхронизирует git и доставляет недостающие silence (например, если
  AM был недоступен в момент создания).
- Подробнее про хранилище — раздел «Git-хранилище» ниже.

## Переменные окружения — backend

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `alert_<имя>` | — | **обязательно ≥1.** URL Alertmanager, каждая = вкладка окружения. Напр. `alert_prod=http://alertmanager.prod.svc:9093` |
| `PORT` | `8000` | порт, который слушает uvicorn внутри контейнера |
| `GIT_REPO_URL` | — | **https**-адрес репы-хранилища правил |
| `GIT_USER` | — | логин для git |
| `GIT_TOKEN` | — | токен с правом **записи** (push) |
| `GIT_BRANCH` | `main` | ветка |
| `GIT_LOCAL_DIR` | `./.hub-repo` | папка локального хранилища (в docker примонтирована в `./data`) |
| `SILENCE_CRON` | `*/5 * * * *` | как часто шедулер проверяет расписания и синхронизирует git |
| `SILENCE_TZ` | `Europe/Moscow` | таймзона, в которой задаётся время окон |
| `AUTH_ENABLED` | `false` | `true` — имя создателя из Keycloak (поле в вебе только показывается); `false` — вводится руками |
| `AUTH_USER_HEADER` | `X-Forwarded-Preferred-Username` | заголовок от oauth2-proxy с именем |
| `AUTH_FALLBACK_USER` | `local` | автор для действий без формы (вкл/выкл/удаление), когда авторизации нет |

`GIT_*` нужны для режима «по расписанию» и истории. Без них (любая из трёх пуста) git
выключается — работаем только локально, шедулер не стартует, а разовый silence в AM поставить
всё равно можно.

Полный пример со всеми переменными и комментариями — [`.env.example`](.env.example) в корне.

## Переменные окружения — frontend

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `BACKEND_URL` | `http://backend:8000` | адрес бэкенда — куда внутренний nginx проксирует `/silences` и `/health` |
| `FRONTEND_PORT` | `80` | порт nginx внутри контейнера |

В dev (`npm run dev`) адрес бэка берётся из `server.proxy` в `vite.config.js`. Фронт всегда
зовёт относительные пути `/silences/...` — адрес бэка в коде не зашит.

## Запуск

### Локальный стенд (dev, без авторизации)

```bash
docker-compose up --build       # http://localhost:8080
```
Поднимет фронт, бэк и тестовый Alertmanager (`.env` в корне). Имя создателя вводится руками.

### С авторизацией через Keycloak

```bash
docker-compose -f docker-compose.yml -f docker-compose.auth.yml up --build
```
Вход через **http://localhost:4180** (Keycloak-логин). Тестовый стенд и инструкция —
[`auth/README.md`](auth/README.md). Прод-настройка — [`auth/PROD.md`](auth/PROD.md).

### Прод

backend и frontend собираются и деплоятся по отдельности из `backend/` и `frontend/`.
Внешний nginx терминирует TLS; авторизация — oauth2-proxy перед вебом (см. `auth/PROD.md`).

## Git-хранилище правил

Правила хранятся как файлы `<env>/<kind>/<id>.json` в git-репе — это «база» с историей:
видно, кто и что менял, можно откатить.

Что нужно: **https**-`GIT_REPO_URL` + `GIT_USER` + `GIT_TOKEN` (с записью) + доступ токену к репе.
Остальное приложение делает само:

- инициализирует папку **на месте** (не `clone`) — не мешает примонтированный README, работает
  и пустая, и существующая репа;
- `origin` всегда чинит на `https+token` — не уходит в ssh;
- пустую/новую репу заводит первым коммитом; существующую — подхватывает;
- при расхождении с удалёнкой **сливает, не теряя локальные правила** (наши файлы приоритетнее),
  переживая даже разные истории;
- каждое действие = отдельный коммит, **автор = пользователь**, в сообщении — кто, что и с каким
  правилом (`<имя> создал/изменил/включил/выключил/удалил <kind> «<name>» (env/id)`).

⚠️ **Не удаляй файлы/папки правил руками в GitLab/GitHub** — приложение синхронизирует репу и
повторит удаление у себя. Удаляй через веб приложения (правило уедет в `old/`, история сохранится).
Под хаб правил лучше отдельная репа.

## Авторизация и кто создаёт правила

- `AUTH_ENABLED=false` — поле «Создатель» вводится руками; в шапке «локальный режим».
- `AUTH_ENABLED=true` — перед вебом стоит oauth2-proxy, без логина в Keycloak не пускает;
  имя берётся из заголовка, поле только показывается, в шапке — имя и «выйти».

В обоих режимах действие записывается автором git-коммита. Детали и безопасность — `auth/PROD.md`.
